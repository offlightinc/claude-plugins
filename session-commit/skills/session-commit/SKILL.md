# Session Commit

현재 대화 세션에서 내가 수정/생성한 파일만 커밋한다.

## 핵심 원칙

- **git diff, git status, git log 등 사전 조회를 하지 않는다.** 세션 중에 어떤 파일을 변경했는지 이미 알고 있다.
- 이 세션에서 변경하지 않은 파일은 절대 포함하지 않는다.
- 바로 `git add <파일들>` + `git commit`만 실행한다.

## 실행 단계

1. 이 세션에서 수정/생성한 파일 목록을 대화 컨텍스트에서 추출한다
2. 해당 파일만 스테이징:
```bash
git add <file1> <file2> ...
```
3. 아래 커밋 메시지 규칙을 따라 커밋 실행

## 커밋 메시지 규칙

### CRITICAL: 커밋 메시지는 무조건 한국어

### 구조: 상단 요약 + 하단 상세

```
<타입>: <제목> (한국어, 3~5줄 이내)

- 핵심 변경 요약 (사람이 빠르게 훑는 용도)

---

<상세 맥락> (AI agent가 검색/히스토리 파악하는 용도, 영어 가능)
- 변경 배경, 의사결정 이유, 영향 범위 등
- 기존에 시도한 접근과 왜 바꿨는지
- 관련 이슈/대화 맥락

🤖 Generated with [Claude Code](https://claude.com/claude-code)

Co-Authored-By: Claude <noreply@anthropic.com>
```

- 상단: 한국어, 간결하게. 섹션 헤더(`## 변경사항` 등) 금지 — 불릿으로 바로 쓴다
- 하단: 자유 형식, AI가 나중에 `git log`로 맥락을 복원할 수 있도록 충분히 작성
- 간단한 변경(오타, 설정 등)은 상단만으로 끝낸다

### 타입별 필수 설명 포맷 (NON-NEGOTIABLE)

#### `feat` — 유저 시나리오 기준으로 설명
- **필수 5요소**: 맥락(언제/왜) → 위치(화면 어디) → 대상(무엇을) → 조작(어떻게) → 결과(무슨 일이)
- 제품을 처음 보는 사람도 커밋 메시지만으로 기능을 그림 그릴 수 있어야 함
- **⚠️ 코드명(컴포넌트명, 함수명, 변수명) 사용 금지** — UI/제품 용어로만 작성
- ❌ `ChatRoomHeader 메뉴에서 Rename 클릭하면 인라인 편집`
- ✅ `채팅방 상단 헤더의 ⋯ 메뉴에서 Rename을 선택하면 제목이 인라인 편집 모드로 전환되고, Enter로 확정하면 이름이 즉시 변경된다`

#### `fix` — 문제 상황 → 해결 구조
- **필수**: 유저가 뭘 했을 때 → 뭐가 안 됐는데 → 어떻게 되도록 고쳤는지
- **⚠️ 코드명 사용 금지** — 증상과 해결을 UI/제품 용어로 설명

#### `refactor` / `chore` / `docs` / `test`
- 기존 포맷 유지 (변경사항 나열)

### 핵심 원칙

- **대화 맥락만 포함**: 현재 세션에서 논의한 내용만 작성
- **⚠️ 가짜 내용 생성 금지**: 논의되지 않은 내용은 절대 작성하지 말 것

## 금지 사항

- `git diff` 실행 금지
- `git status` 실행 금지
- `git log` 실행 금지
- 세션에서 변경하지 않은 파일 포함 금지
